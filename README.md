**Техническое задание**

## 1. ВВЕДЕНИЕ

Наименование разработки: веб-сервис для автоматизированной оценки юзабилити веб-сайтов «UX Analyzer».

Краткая характеристика области применения: сервис предназначен для анализа пользовательских интерфейсов веб-сайтов с целью выявления нарушений эвристик юзабилити Якоба Нильсена и требований стандарта доступности Web Content Accessibility Guidelines (WCAG) 2.1.

Объект, в котором используется программа: система автоматизированной проверки качества веб-интерфейсов, предназначенная для малого и среднего бизнеса, веб-студий, маркетинговых агентств и владельцев сайтов.

## 2. ОСНОВАНИЯ ДЛЯ РАЗРАБОТКИ

Настоящая разработка инициирована автором в ответ на выявленную потребность в автоматизированных инструментах оценки юзабилити веб-ресурсов. Проект выполняется в рамках исследовательской работы с целью последующего промышленного внедрения.

Тема разработки: «Разработка веб-сервиса анализа пользовательской структуры сайта с оценкой юзабилити и генерацией UX-рекомендаций».

## 3. НАЗНАЧЕНИЕ РАЗРАБОТКИ

Функциональное назначение: сервис обеспечивает автоматический анализ веб-страниц, выявление нарушений юзабилити, расчёт интегрального UX-скора (0–100), генерацию текстовых рекомендаций по улучшению интерфейса и визуализацию нарушений на скриншоте страницы.

Эксплуатационное назначение: сервис предназначен для использования владельцами веб-сайтов, менеджерами продуктов, маркетологами, веб-разработчиками и специалистами по доступности, не требующими специальных технических знаний для интерпретации результатов.

## 4. ТРЕБОВАНИЯ К ПРОГРАММЕ ИЛИ ПРОГРАММНОМУ ИЗДЕЛИЮ

### 4.1. Требования к функциональным характеристикам

Программа должна выполнять следующие функции:

- регистрацию и аутентификацию пользователей с использованием JWT-токенов и хэширования паролей
- создание, редактирование, просмотр и удаление проектов (групп анализов) с уникальным текстовым идентификатором
- запуск анализа веб-страницы по указанному URL
- сбор DOM-структуры через headless-браузер, включая координаты, текст, размеры и атрибуты доступности всех интерактивных элементов
- сохранение полноэкранного скриншота анализируемой страницы
- расчёт 34 метрик, сгруппированных в 11 категорий
- вычисление интегрального UX-скора (0–100) и категорийных оценок с учётом весов метрик
- генерацию текстовых рекомендаций по устранению нарушений через большую языковую модель
- визуализацию нарушений на скриншоте страницы в виде цветных прямоугольников с всплывающими подсказками
- предоставление интерактивного дашборда с радарными и столбчатыми диаграммами

### 4.2. Требования к надежности

Программа должна обеспечивать устойчивое функционирование при одновременной обработке до 5 параллельных анализов без деградации производительности.

При сбоях на этапе сбора данных или анализа должны предусматриваться повторные попытки с логированием ошибок.

Время восстановления после отказа не должно превышать 30 секунд при перезапуске сервиса.

### 4.3. Условия эксплуатации

Требуется наличие подключения к сети Интернет для доступа к анализируемым сайтам и вызова LLM API.

Обслуживание осуществляется одним администратором без постоянного присутствия.

### 4.4. Требования к составу и параметрам технических средств

Минимальные требования к техническим средствам для развёртывания сервера:

- Процессор: 4 ядра, 2.5 ГГц
- Оперативная память: 8 ГБ (рекомендуется 16 ГБ)
- Дисковое пространство: 20 ГБ
- Сеть: доступ в Интернет, статический IP (опционально)
- Операционная система: Ubuntu 22.04, Windows Server 2019/2022, или macOS

Для запуска в режиме разработки допустимо использование рабочей станции с аналогичными характеристиками.

### 4.5. Требования к информационной и программной совместимости

Информационные структуры на входе: программа принимает URL веб-страницы через REST API.

Информационные структуры на выходе: программа возвращает данные в форматах JSON, PNG, а также отображает информацию через веб-интерфейс.

Требования к исходным кодам и языкам программирования:

- Бэкенд: Python 3.11+ (FastAPI, SQLAlchemy, Playwright, Pillow)
- Фронтенд: React 18+ (JavaScript/JSX, CSS)
- База данных: PostgreSQL 16+
- Кэш и очередь: Redis 7+
- Контейнеризация: Docker, Docker Compose

Требования к программным средствам:

- Веб-сервер: Uvicorn для запуска FastAPI
- Статические файлы (скриншоты): монтируемая директория или S3-совместимое хранилище

### 4.6. Требования к пользовательскому интерфейсу (UX/UI)

| Требование | Описание |
|------------|----------|
| Адаптивность | Интерфейс должен корректно отображаться на экранах с разрешением от 1280×720 до 1920×1080, а также на мобильных устройствах (планшетах) |
| Интерактивность | Дашборд должен содержать интерактивные элементы: фильтры нарушений по категории и критичности, переключение между анализами, масштабирование оверлея |
| Визуализация нарушений | Оверлей нарушений должен отображаться поверх скриншота страницы в виде цветных прямоугольников: красный — critical, оранжевый — warning, синий — info |
| Подсказки | При наведении на прямоугольник оверлея должна отображаться всплывающая подсказка с текстом нарушения и рекомендацией |
| Доступность | Интерфейс должен соответствовать базовым критериям доступности: контрастность текста, навигация с клавиатуры, поддержка скринридеров |

### 4.7. Требования к обработке данных и метрикам

| Требование | Описание |
|------------|----------|
| Состав метрик | Система должна поддерживать не менее 34 метрик, сгруппированных в 11 категорий (видимость, читаемость, навигация, консистентность, формы, макет, доступность, производительность, обработка ошибок, справка и поддержка, WCAG) |
| Конфигурируемость метрик | Метрики должны быть описаны в JSON-конфигурации. Добавление новой метрики не должно требовать изменения программного кода |
| Типы метрик | Должны поддерживаться три типа метрик: boolean (0/1), ratio (0..1), score (0..100) |
| Веса метрик | Каждая метрика должна иметь вес (0.5–1.0), влияющий на итоговый UX-скор |
| Пороги нарушений | Для метрик типа ratio и score должен быть задан порог нарушения (по умолчанию 0.8). Для boolean — нарушение при значении 0 |
| Хранение результатов | Значения всех метрик должны сохраняться в БД для последующего анализа и отладки |
| Нормализация | При агрегации оценки категорий значения метрик должны нормализоваться к шкале 0–100 (boolean/ratio умножаются на 100, score остаются без изменений) |

### 4.8. Требования к безопасности и хранению данных

| Требование | Описание |
|------------|----------|
| Аутентификация | Доступ к API должен осуществляться по JWT-токенам с временем жизни не менее 30 минут |
| Хэширование паролей | Пароли пользователей должны храниться в БД в хэшированном виде (bcrypt) |
| Разграничение доступа | Пользователь имеет доступ только к своим проектам и анализам |
| CORS | Должна быть настроена CORS-политика, разрешающая запросы с доверенных доменов фронтенда |
| Логирование действий | Система должна логировать все значимые действия пользователей (регистрация, вход, запуск анализа, удаление проекта) |
| Очистка данных | URL анализируемых страниц не должны храниться в БД дольше 30 дней (если иное не настроено пользователем) |
| Защита от несанкционированного доступа | Все эндпоинты, кроме /health, /docs, /users/register, /users/login, должны требовать JWT-токен |

### 4.9. Требования к масштабируемости и производительности

| Требование | Описание |
|------------|----------|
| Горизонтальное масштабирование | Сервисы Collector Worker и Analysis Worker должны поддерживать запуск нескольких реплик для распределения нагрузки |
| Очереди задач | Взаимодействие между микросервисами должно осуществляться через асинхронные очереди на базе Redis Streams |
| Кэширование | DOM-элементы страницы должны временно храниться в Redis (TTL 1 час) для разгрузки БД |
| Параллельные анализы | Система должна поддерживать одновременную обработку не менее 5 анализов без деградации производительности |
| Время отклика API | API Gateway должен отвечать на запросы (без учёта фоновых задач) не более 200 мс |
| Время сбора данных | Сбор данных одной страницы не должен превышать 30 секунд |
| Время анализа | Вычисление метрик для одной страницы не должно превышать 10 секунд |

Защита информации: аутентификация по JWT, хэширование паролей (bcrypt), разграничение доступа к проектам на уровне пользователя, CORS-политика для взаимодействия с фронтендом.

## 6. ТРЕБОВАНИЯ К ИНТЕГРАЦИИ С ВНЕШНИМИ СИСТЕМАМИ

| Требование | Описание |
|------------|----------|
| LLM API | Система должна интегрироваться с OpenRouter API (или совместимым) для генерации рекомендаций |
| API документация | Все REST-эндпоинты должны быть документированы через OpenAPI (Swagger) по адресу /docs |

| Коды ответов | Должны использоваться стандартные HTTP-коды: 200, 201, 400, 401, 403, 404, 422, 500 |
| Пагинация | Эндпоинты, возвращающие списки (проекты, анализы, нарушения), должны поддерживать параметры skip и limit |

## 7. ТРЕБОВАНИЯ К МОНИТОРИНГУ И ЛОГИРОВАНИЮ

| Требование | Описание |
|------------|----------|
| Логирование ошибок | Все ошибки (исключения, таймауты, недоступность LLM) должны записываться в логи с указанием времени и контекста |
| Уровни логирования | Должны поддерживаться уровни: DEBUG, INFO, WARNING, ERROR |
| Метрики производительности | Должны собираться метрики: время загрузки страницы, время анализа, количество параллельных анализов |
| Мониторинг очередей | Должна быть возможность отслеживать длину очередей collector_queue и analyzer_queue |
| Алертинг | При превышении времени ответа LLM API или при падении воркера должно формироваться предупреждение (в лог/email) |

## 8. ТЕХНИКО-ЭКОНОМИЧЕСКИЕ ПОКАЗАТЕЛИ

Ориентировочная экономическая эффективность: программа позволяет сократить расходы на проведение UX-аудита на 80–95% по сравнению с привлечением внешних экспертов (средняя стоимость профессионального аудита одного сайта составляет от 50 000 до 500 000 рублей). Предполагаемая годовая потребность — от 100 до 1000 анализов при условии обслуживания 20–200 активных проектов.

Экономические преимущества: автоматизация процесса, снижение времени анализа с 2–5 дней до 5–10 минут, возможность регулярных проверок (ежедневно/еженедельно) без дополнительных затрат, наглядная визуализация результатов, понятная для нетехнических специалистов.

## 10. ПОРЯДОК КОНТРОЛЯ И ПРИЕМКИ

Виды испытаний:

- Модульное тестирование каждого компонента
- Интеграционное тестирование взаимодействия микросервисов и очередей
- Функциональное тестирование на 10+ реальных веб-сайтах различной сложности
- Тестирование производительности (нагрузка до 5 параллельных анализов)
- Тестирование удобства использования с участием фокус-группы

Общие требования к приемке: программа считается принятой при успешном прохождении всех видов испытаний, соответствии функциональным требованиям, оформлении комплекта программной документации и развёртывании рабочего экземпляра на целевой инфраструктуре.
